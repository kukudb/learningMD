# Cocos Creator + TypeScript 学习进度与教学规则

## 一、教学规则（适用于所有后续学习）

> 学习者背景：有 Java 基础的程序员

### 规则 1：先提供详尽教学内容，不立即布置练习
- 每节知识点，我会给出**完整的、自洽的**教程，包括：
  - 语法说明（与 Java 的异同点）
  - 核心概念解释
  - 至少 3 个**不同场景**的示例代码（示例之间差异明显，不能只改变量名）
- 学习者阅读后，可以随时提出疑问（比如”这个和 Java 的 interface 有什么区别？”）
- 只有在确认学习者**无疑问、理解透彻**后，才会进入下一步。
- **大节拆分**：每大节的教学内容要划分成多个小节，由浅入深。每次只给出一小节的内容，不得一次性抛出整节所有内容。一小节讲完后，按规则 4 的互动确认机制决定是否推进到下一小节。

### 规则 2：练习与教学示例**不能高度相似**
- 教学示例演示的是**基础用法**，但练习会要求：
  - 组合多个知识点
  - 解决一个略有变化或逆向思维的问题
  - 或是修补一段有缺陷的代码（而不是从零模仿）
- **禁止**直接复制示例代码就能完成练习。练习必须迫使学习者做额外思考。

### 规则 3：利用 Java 基础进行类比教学
- 每次讲解 TypeScript 特性时，我会主动对比 Java 中的对应概念：
  - 例如：接口（`interface`）vs Java 的接口、类型收窄 vs `instanceof`、泛型擦除 vs 运行时泛型等。
- 突出 TypeScript 作为**结构化类型系统**与 Java 名义类型的区别。

### 规则 4：互动确认后再推进
- 在每个小节结束时，我会主动问：“关于 XX 知识点，你还有疑问吗？如果没有，我将给出本节练习。”
- 在你明确表示“可以开始练习”之后，我才会发布练习题目。

### 规则 5：增加“代码诊断”型练习
- 部分练习会以**有缺陷的代码**形式给出，你需要：
  - 指出类型错误或逻辑错误
  - 修正代码并说明原因
- 这种形式能检验你是否真正理解概念，而非机械模仿。

### 规则 6：对学习过程中的疑问归纳入库
- 学习过程中产生的疑问及解答，将整理到 `cocos学习疑问解答.md` 文档中
- 按照对应知识章节分类存放，方便回顾翻阅
- 每个问答保留原始提问角度和解答要点
- **相似疑问合并**：同一知识点衍生出的多个追问，应归纳为一个条目。每条保留三要素：① 提问者的原始视角或思维定式（如"从 Java 多线程模型出发的预期"）② 暴露出的概念混淆点 ③ 核心原理的准确解释。不保留：重复表述、逐轮追问的原话、已被后续追问覆盖的中间结论。目的：精简知识库，翻看时快速定位理解偏差，不被重复内容分散注意力。

### 规则 7：使用对比表格辅助教学
- 对于容易混淆的概念（如 `type` vs `interface`、`abstract class` vs `interface`），我会提供**对比表格**。
- 必要时，练习中会要求你补充表格中的缺失项，以加深理解。

### 规则 8：归档学习者的概念混淆点与易错模式
- 每次问答中学习者暴露的**概念混淆**，以及练习中反复出现的**易错模式**，随学习进度归档到 `cocos学习疑问解答.md`。
- 归档格式：在对应章节末尾增加「概念混淆点」和「练习易错模式」二级段落，简要记录现象和根因。
- 目的：后续复习时快速定位自己的薄弱环节，避免重复踩坑。

### 规则 9：教学完毕后自动提交 Markdown 文档到 Git
- 每次教学结束后，完成以下流程：
  1. 总结并更新 `cocos学习进度.md`（进度、掌握程度、下一步行动）
  2. 如有新增疑问或概念混淆，同步更新 `cocos学习疑问解答.md`
- 文档更新完成后，**自动执行 Git 操作**：
  1. 仅暂存本次教学中实际修改的 `.md` 文件（不提交 `.ts` 练习文件等）
  2. Commit 到 `master` 分支，commit 信息格式：`<本次学习内容简述>；如有文档修复或目标更新，用分号追加说明`
  3. Push 到 `origin` 的 `master` 分支
- 目的：学习进度和知识沉淀与代码仓库同步，方便随时回溯学习历程。

- [x] 回调地狱（Callback Hell）的成因
- [x] setTimeout 执行时机：调用时立即计时，回调进入宏任务队列
- [x] setTimeout 的 delay 是"最小保证"，不是精确时间
- [x] 宏任务（MacroTask）的定义与来源（script整体、setTimeout、I/O、事件回调）
- [x] 微任务（MicroTask）的定义与来源（Promise.then、queueMicrotask、await 后续代码）
- [x] 事件循环的精确算法：取一个宏任务 → 执行 → 清空所有微任务 → 渲染 → 下一轮
- [x] 微任务优先于宏任务的原因：保证 Promise 回调在本轮完成，不被新宏任务插队
- [x] 事件循环不是另一个线程——它就是主线程在调用栈清空后的运行方式
- [x] JS 代码执行是 100% 同步的——"异步"描述的是时间解耦，不是执行并发
- [x] 定时器不可靠的根因：回调回到主线程必须排队，与"在哪个线程定时"无关
- [x] deltaTime 补偿模式：不追求准时调用，只追求逻辑结果与真实时间一致
- [x] JS 中定时任务的三种写法（setInterval / 递归 setTimeout / 递归 + 时间戳纠偏）
- [x] 空闲时 setTimeout 误差 ±1~4ms；阻塞时误差可达百毫秒级
- [x] 主线程铁律：避免阻塞操作，重活拆碎或扔给 Worker
- [x] async/await 本质：编译器将 async 函数沿 await 切成多段，每段是 .then() 微任务
- [x] await "暂停"的真相：函数交出控制权（栈帧弹出），剩余代码包进 .then() 回调
